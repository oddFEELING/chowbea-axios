# Security Policy

## Reporting a Vulnerability

If you believe you've found a security vulnerability in `chowbea-axios`, please report it **privately** rather than opening a public issue.

**Preferred channel:** open a private security advisory on GitHub:

> https://github.com/oddFEELING/chowbea-axios/security/advisories/new

Alternatively, email **platforms@chowbea.com** with the subject line `[security] chowbea-axios`.

Please include:

- A description of the issue and its potential impact
- Steps to reproduce, ideally with a minimal proof-of-concept
- The version of `chowbea-axios` affected (e.g. `2.0.0-alpha.21`)
- Your assessment of severity, if known

### Response timeline

| Stage | Target |
|---|---|
| Acknowledgement | **Within 7 days** of receiving the report |
| Initial triage and severity assessment | Within 14 days |
| Fix released or mitigation published | Best effort — typically within 30 days for High/Critical; longer for low-severity issues that require breaking changes |
| Public disclosure | Coordinated with the reporter, usually once a fix is available on the latest supported alpha |

We will keep you informed throughout the process. If we determine the issue is out of scope (e.g. it affects a dependency we don't ship, or a configuration the user controls), we will explain why.

## Supported Versions

| Version | Supported |
|---|---|
| Latest `2.x` (published as `latest`) | ✅ |
| Older `2.x` releases | ❌ — fix forward by upgrading |
| `2.0.0-alpha.*` prereleases | ❌ — superseded by `2.0.0` stable |
| `1.x` and earlier | ❌ |

We support **only the latest `2.x` release** on the `latest` dist-tag. Security fixes are released as patch versions (e.g. `2.0.1`); older `2.x` releases do not receive backports. The recommended response to a security advisory is to upgrade to the latest `2.x`.

When `3.x` ships, this policy will be revisited to add a defined support window for `2.x`.

## Scope

In-scope:

- The published `chowbea-axios` CLI and its runtime (parsing user-controlled OpenAPI specs, generating TypeScript code, writing to the filesystem)
- The TypeScript code generated by the CLI (e.g. injection or type-safety issues introduced at codegen time)
- The Vite plugin templates exported via `chowbea-axios/vite`
- The CI template shipped under `templates/`

Out of scope:

- Vulnerabilities in user-supplied OpenAPI specs themselves (we sanitize what we emit, but we don't validate spec correctness beyond what's needed for generation)
- Configurations the user explicitly opts into (e.g. an `output.folder` pointing at a sensitive directory)
- Vulnerabilities in transitive dependencies — please report those upstream first; we will track the advisory and bump pinned versions as fixes become available
- Issues affecting end-of-life or unsupported versions (see table above)

## Acknowledgements

Reporters who follow this policy will be credited in the release notes for the fix (unless they request anonymity).
